# Evaluator Rubric

评审评分表。会话结束后或到里程碑时，用它评估 agent 做的东西够不够格。

## 六个维度（每项 0-2 分）

### 1. 正确性
- 0 分：行为与目标完全不符
- 1 分：部分正确，有明显缺陷
- 2 分：行为完全符合目标功能

### 2. 验证
- 0 分：没有任何验证被执行
- 1 分：部分验证通过，但有遗漏
- 2 分：所有要求的验证都跑过并留下证据

### 3. 范围纪律
- 0 分：严重超出范围，引入无关改动
- 1 分：轻微越界，但不影响核心功能
- 2 分：严格保持在选定功能范围内

### 4. 可靠性
- 0 分：重启后行为不一致
- 1 分：基本可用，但边界情况有问题
- 2 分：重启或重跑后行为稳定

### 5. 可维护性
- 0 分：代码混乱，无文档
- 1 分：代码可读，但文档不完整
- 2 分：代码清晰，文档完整，可直接交给下一轮会话

### 6. 交接准备度
- 0 分：新会话需要大量摸索才能继续
- 1 分：大部分信息可用，但有缺失
- 2 分：新会话只靠仓库内文件就能无缝继续

## 结论选项

- **Accept** — 达标（总分 ≥ 8）
- **Revise** — 需要修补才能接受（总分 5-7）
- **Block** — 有根本性问题，需要先解决（总分 < 5）

## 校准说明

Evaluator 需要校准。开箱即用的 agent 做评审很弱——它会发现问题，然后把自己说服到通过。你需要反复校准：

1. 用 evaluator 给一个已完成的 sprint 打分
2. 把它的分数和你自己的人工判断对比
3. 有分歧的地方，把 rubric 里的通过/失败标准写得更具体
4. 对同一个输出重新跑 evaluator，看对齐了没有
5. 重复直到 evaluator 的判断和人工评审基本一致

预计需要 3-5 轮校准。
